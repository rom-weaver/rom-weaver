rom–weaver

 patch layout prototype

A rethink of the apply workflow's ROM + patch cards: no drawers — every value is on the table. One in/out ledger per card, a verified-ROM identity band, version + author on the patch byline, and one pencil that edits name · version · author · description together. The Stack · Hug · Split · Table switcher up top swaps between four desktop layout candidates; toggle Notes for annotations.

What changed

  1. 1 Verified ROM identity (future field): the database match — proper title, region, revision, serial — rides the ROM card as a stitched band.
  2. 2 The ledger replaces the Checks drawer: always open, input and output aligned per algorithm. A “—” cell is honest — the patch doesn't declare it; click it to add your own expected value.
  3. 3 Provenance is visible per value: ° built into the patch file, ✎ added by you (removable).
  4. 4 Version + author are first-class patch metadata on the byline, under the display name.
  5. 5 One pencil, one form: name, version, author, description edited together — replaces the two separate pencils.
  6. 6 Endpoint chips carry the chain semantics: the first enabled patch checks the ROM, the last one sets the expected result; mid-chain checks are recorded only.
  7. 7 The expected result is its own stub card, fed by the chain-output checks — where “verified after weaving” lands.
  8. 8 ROM checksums always visible too, one column per variant (as dropped / header stripped) — the database match mark sits on the variant that matched.
